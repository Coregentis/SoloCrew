# SOLOCREW-REPO-V1-VS-OPERATIONAL-V1-BOUNDARY-v0.1

## Purpose

This baseline formally separates:

- current SoloCrew repo/platform `v1.0`
- later SoloCrew Operational V1.0

It exists so the current lawful repo-level `v1.0` closure can remain true
without being over-read as runtime completeness, execution authority, or full
operational closure.

## Current Repo / Platform `v1.0`

Current repo/platform `v1.0` is:

- `v1.0-solo-operator-platform delivered and closed`

That means the current repository lawfully has, in bounded form:

- a stable single-cell operating core
- a bounded portfolio / Secretary layer
- a lawful business-pack / metrics-pack mount model
- coherent management, organization, execution-adjacent, and memory/evidence
  planes
- a bounded platform summary / delivery-readiness layer
- bounded runtime-dependent downstream truth hardening over current upstream
  truth
- explicit upstream boundary discipline preserved

## What Current Repo / Platform `v1.0` Does Not Mean

Current repo/platform `v1.0` does **not** mean:

- direct approve / reject / dispatch / execute behavior now exists
- provider or channel execution now exists
- SoloCrew owns runtime authority
- SoloCrew owns protocol authority
- SoloCrew has become a full runtime repository
- bounded downstream truth has become workflow law
- bounded downstream truth has become delivery authorization
- Operational V1.0 closure has already been achieved

## Operational V1.0

Operational V1.0 is a later closure class.

It should be read as:

- a founder-usable one-person-company operating system in practice, not only a
  bounded repo/platform baseline in governance terms

Operational V1.0 is therefore downstream of current repo/platform `v1.0`.
It may use the current repo/platform line as its starting platform baseline,
but it is not identical to it.

## Operational V1.0 Closure Conditions

Operational V1.0 should not be closed unless all of the following are true:

1. `Cognitive_OS` provides a first lawful runtime-side continuity substrate that
   is stronger than the current bounded in-memory/minimal-loop baseline.
2. `Cognitive_OS` provides a first lawful governed-activation path stronger than
   the current bounded minimal activation scaffold.
3. `Cognitive_OS` provides a first lawful semantic-truth / impact path covering
   delta-intent, drift, conflict, and bounded impact interpretation beyond the
   current minimal executable baseline.
4. SoloCrew provides a founder-facing exception plane with evidence-backed
   review posture rather than explanatory shell posture alone.
5. SoloCrew can run at least one real bounded business loop from input through
   review and bounded action to evidence and outcome.
6. All of the above remain downstream of:
   - `MPLP Protocol`
   - `Cognitive_OS`
   and do not collapse runtime or protocol authority into product space.

## Current Boundary Judgment

Current truthful judgment is:

- repo/platform `v1.0`: `YES`
- Operational V1.0: `NO`

That is not a contradiction.
It is the required current boundary.

## Governance Rule

Future work must therefore preserve all of the following at once:

- keep current repo/platform `v1.0` closure intact
- refuse semantic over-read from repo/platform closure into operational closure
- refuse product-side invention of missing runtime law
- refuse downstream pressure as a basis for protocol-law widening

## Practical Reading Rule

When SoloCrew says `v1.0` in current repo-governance language, the default read
must be:

- repo/platform `v1.0`

When future work claims `Operational V1.0`, it must first satisfy the closure
conditions in this baseline and be governed as a separate later line.
