# SOLOCREW-VERSION-ROADMAP-v0.1

## Purpose

This document freezes the post-`v0.1` version sequence for SoloCrew so structural work proceeds in coherent waves instead of feature drift.

## Version Sequence

### `v0.1-baseline`

Goal:

- establish SoloCrew as a truthful downstream baseline over bounded `Cognitive_OS` runtime surfaces

Mandatory baseline:

- repo initialization and authority-order freeze
- projection, DTO, and workflow baseline documents
- bounded runtime-consumption contract
- projection assembly baseline over bounded upstream truth
- runtime session entry and continuity/reload truth tests
- formal `v0.1` release note and closure record

Explicitly deferred:

- structural constitution expansion
- usable product-page implementation
- multi-cell behavior
- Secretary behavior
- provider execution
- channel integrations

Goal:

- freeze the lawful multi-cell foundation between sealed `v0.3` and future `v0.5-portfolio-secretary-beta`

Current status:

  - `v0.1` through `v0.4` version-line meaning

Explicitly deferred:

- growth or BD pack runtime execution
- provider execution

Current status:

Explicitly deferred:

### `v1.0-solo-operator-platform`

Goal:

- deliver SoloCrew as a coherent solo-operator business platform with stable structural layers

Mandatory baseline:

- stable single-cell operating core
- bounded portfolio/Secretary layer
- lawful business-pack and metrics-pack mount model
- coherent management, organization, execution, and memory/evidence planes
- explicit upstream boundary discipline preserved

Explicitly deferred:

- anything that would collapse product, runtime, and protocol authority into one layer
- speculative expansion that has not survived blocker-ledger review

## Roadmap Rule

Each version must preserve:

- `MPLP Protocol -> Cognitive_OS -> SoloCrew`
- one coherent wave at a time
- no silent promotion of product structure into runtime or protocol authority

## Version Semantics Rule

In this repo:

- the release line is the authoritative product-governance version label
- the private `package.json` version is a workspace/tooling label only

So the introduction of `v0.4-multi-cell-foundation` changes the roadmap and governance truth.
It does not by itself require changing the private package version or claiming a publishable package release.

`v0.4` now also carries a formal closure record and boundary-clarification pack.
That closure pack does not widen the line into `v0.5`.
