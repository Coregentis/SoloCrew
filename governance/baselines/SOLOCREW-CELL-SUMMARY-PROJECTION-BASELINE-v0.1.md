# SOLOCREW Cell Summary Projection Baseline v0.1

## Purpose

This baseline freezes the minimum lawful `Cell Summary Projection` for `v0.4-multi-cell-foundation`.

It defines how one `Cell` may be compressed into a bounded top-level summary unit without pretending that summary is runtime law or a complete management cockpit.

## Summary Role

A `Cell Summary Projection` is:

- a product-facing summary unit
- derived from bounded SoloCrew product truth
- suitable for future top-level inspection, listing, and focus selection

A `Cell Summary Projection` is not:

- a mother-runtime object
- a runtime-backed orchestration law
- a complete management dashboard
- a substitute for entering the `Cell Console`

## Minimum Summary Unit

The minimum lawful summary unit should expose:

- `cell_identity`
  - enough to identify which `Cell` is being summarized
- `cell_charter_headline`
  - one bounded framing line about the `Cell` purpose
- `current_objective_headline`
  - one bounded statement of the active objective posture
- `delivery_posture`
  - high-level delivery state only
- `work_posture`
  - bounded active versus blocked posture only
- `continuity_note`
  - bounded continuity/reload truth only
- `attention_signal`
  - whether the `Cell` likely needs review, approval, or intervention
- `deferred_surface_note`
  - explicit reminder that summary is not complete control truth

## Summary Boundary

The summary boundary is strict:

- summary is inspect-oriented
- summary is not business-control complete
- summary does not carry lawful authority to rewrite internal `Cell` policy or runtime state
- summary does not silently become a `Cognitive_OS` summary runtime surface

If a future management surface needs deeper control, it must still pass through lawful product contracts and later runtime blocker review where necessary.

## Current Owning Layer

Today the `Cell Summary Projection` belongs to:

- SoloCrew product projection space

It may stay there while summary remains:

- bounded
- read-oriented
- product-facing
- reconstructable from already lawful SoloCrew truth

## Later Upstream Blocker Trigger

`Cell Summary Projection` becomes a legitimate later `Cognitive_OS` blocker-intake candidate only if SoloCrew needs:

- durable runtime-backed summary generation across many `Cell`s
- reusable runtime summary policies
- shared summary state that cannot stay truthful as downstream product projection

This document does not promote that blocker by itself.

## Explicit Deferrals

This baseline explicitly defers:

- Secretary card UX
- portfolio dispatch behavior
- broad KPI rollups
- runtime-complete cross-cell orchestration
- enterprise reporting

## Baseline Rule

If a future summary field mainly exists to support:

- portfolio dispatch control
- enterprise reporting
- runtime law
- broad analytics

then it does not belong in the minimum `v0.4` summary baseline by default.
