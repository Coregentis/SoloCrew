# SOLOCREW Secretary Interface UX Baseline v0.1

## Purpose

This document freezes the first lawful UX shape for the `v0.5-portfolio-secretary-beta` surface.

It is a UX baseline only.
It does not claim implementation.

## Secretary Beta Entry

The first lawful Secretary beta UX must provide:

- one top-level portfolio entry
- one Secretary-facing portfolio view distinct from per-cell views
- bounded cell cards, queues, or status shelves
- bounded portfolio-level posture summaries over multiple cells

The entry must remain recognizably above the cell layer without claiming ownership of runtime law.

## Bounded Secretary View Versus Cell View

The UX must preserve a visible separation between:

- Secretary view
  - top-level portfolio posture
  - cross-cell management visibility
  - bounded queue and review posture
- cell view
  - cell-local detail, continuity, and management inspection
  - cell-local operational context
  - bounded downstream product projection over runtime-private truth

The Secretary beta surface may coordinate with cells.
It must not erase the distinction between Secretary and cell roles.

## What The Secretary Can See

The first lawful Secretary beta UX may show:

- a list of cells as cells
- cell summary cards
- queue or review shelves
- bounded status signals
- bounded management object visibility
- bounded review, approval, or delivery posture summaries
- explicit non-claims and deferred surfaces

## What The Secretary May Initiate

The first lawful Secretary beta UX may initiate only bounded product-level actions such as:

- selecting a cell
- opening a cell view
- preparing a bounded handoff or review context
- placing bounded intent into a queue or review posture
- requesting bounded next-step attention from a cell

These are beta-level product initiations only.
They are not direct execution rights.

## What Still Requires Handoff

The following must still require bounded downstream handoff rather than direct Secretary execution:

- cell-specific management context
- cell-specific review context
- cell-specific delivery follow-up context
- runtime-backed state changes
- provider-backed execution
- channel entry or handoff

## What Must Stay Out Of The UX

The first lawful Secretary beta UX must not imply:

- full runtime-complete portfolio control
- direct provider execution
- direct channel operation
- enterprise team collaboration
- deep permission management
- autonomous operating-company behavior
- protocol authority

## UX Rule

If a proposed Secretary beta UX mainly behaves like:

- an enterprise operating cockpit
- a dispatch center with direct execution power
- a broad analytics suite
- a runtime-administration console

then it is not a valid `v0.5` Secretary UX baseline.

If it behaves like the first bounded top-level management view over multiple cells while preserving explicit handoff and boundary semantics, it fits this baseline.
