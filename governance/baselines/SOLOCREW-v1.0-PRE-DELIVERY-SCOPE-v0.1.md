# SOLOCREW v1.0 Pre-Delivery Scope v0.1

## Purpose

This document defines the lawful pre-delivery scope between the current
`v0.5-portfolio-secretary-beta` state and eventual
`v1.0-solo-operator-platform` delivery.

It exists so future work can move toward `v1.0` without silently:

- widening `v0.5`
- collapsing product and runtime authority
- promoting product semantics into protocol law
- smuggling direct-control behavior into near-term planning

## Current Starting Point

Current SoloCrew truth is:

- one sealed bounded single-cell core
- one sealed governed multi-cell foundation
- one reopened-and-landed bounded explanatory `v0.5` beta lane

This is not yet formal `v1.0` delivery.

## Governing Rule

Pre-delivery work must improve delivery readiness without changing authority
ownership.

The required authority order remains:

- `MPLP Protocol -> Cognitive_OS -> SoloCrew`

## Allowed Pre-Delivery Work

The following work classes are allowed before formal `v1.0` delivery:

- structural product-coherence work that makes current bounded surfaces read as
  one platform rather than as isolated version lanes
- lawful business-pack and metrics-pack mount-model clarification, surfacing,
  and bounded downstream interpretation
- cross-plane alignment work across management, organization, execution, and
  memory/evidence product planes
- explanatory and evidence-linked product hardening that remains downstream and
  omission-aware
- bounded platform-summary and delivery-readiness framing that stays
  non-executing
- governance, gap-matrix, sequencing, audit, and release-gate work needed to
  decide whether `v1.0` can be claimed
- runtime-dependent downstream consumption only when current `Cognitive_OS`
  truth already supports it and does not require SoloCrew to invent upstream
  law

## Forbidden Pre-Delivery Work

The following work remains forbidden unless separately re-governed:

- direct approve behavior
- direct reject behavior
- direct dispatch behavior
- direct execute or provider/channel behavior
- workflow-engine semantics
- runtime authority ownership inside SoloCrew
- protocol authority ownership inside SoloCrew
- product-side imitation of upstream workflow-truth objects that `Cognitive_OS`
  has not adopted
- silent conversion of `v0.5` packet/posture semantics into upstream workflow
  law
- enterprise, team, multi-tenant, or autonomous-company pull-forward

## What Must Remain Non-Executing

Unless separately re-governed, the following must remain non-executing even in
near-term planning:

- Secretary and portfolio management surfaces
- handoff and review packet posture semantics
- rationale, evidence, provenance, and omission-aware narration
- business-pack and metrics-pack mount treatment
- platform-summary and readiness framing

These may become clearer and more coherent.
They may not become execution authority by convenience.

## What Counts As Scope Creep

The following count as scope creep or authority collapse by default:

- adding control affordances that turn a product explanation surface into an
  action surface
- widening a structural mount into execution-bearing pack behavior without a
  separate gate
- claiming runtime ownership because SoloCrew displays richer evidence or
  posture
- treating repeated product usage as automatic justification for MPLP or
  `Cognitive_OS` promotion
- building broad platform breadth before the mount model and plane coherence are
  settled

## Near-Term Pre-Delivery Focus

The lawful near-term focus is:

- make the current product structure cohere into a delivery-grade platform
  baseline
- do that without adding direct-control semantics
- do that without rewriting current upstream truth

## Pre-Delivery Scope Judgment

The truthful pre-delivery scope between current `v0.5` and eventual `v1.0` is:

- bounded structural work
- bounded explanatory work
- bounded runtime-dependent downstream consumption where upstream truth already
  exists
- governance and gate work

It is not:

- execution expansion
- workflow-authority expansion
- protocol-authority expansion
- silent promotion of `v0.5` into `v1.0`
