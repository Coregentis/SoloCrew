# SOLOCREW v0.4 Multi-Cell Foundation Scope v0.1

## Purpose

This document freezes `v0.4-multi-cell-foundation` as the missing intermediate version line between sealed `v0.3-single-cell-usable` and future `v0.5-portfolio-secretary-beta`.

`v0.4` is a governance-first and boundary-tightening line.
It is not a broad product-code expansion wave.

## Why v0.4 Exists

Sealed `v0.3` already gives SoloCrew:

- one actual bounded single-cell operator console line
- bounded correction/review and correction/apply truth
- honest continuity and reload truth

Future `v0.5` is reserved for:

- bounded Secretary behavior
- bounded portfolio behavior
- later cross-cell management-plane expansion

So a missing intermediate line is needed to freeze the lawful multi-cell foundation before Secretary beta begins.

## v0.4 Goal

The goal of `v0.4-multi-cell-foundation` is to freeze:

- lawful multi-cell coexistence
- cell summary projection
- the minimum management object family
- what remains product-projected versus what later becomes an upstream runtime blocker

`v0.4` is a precondition line for future `v0.5-portfolio-secretary-beta`.

## Must Be Included

`v0.4-multi-cell-foundation` must include:

- a lawful product-level statement that multiple `Cell`s may coexist and be listed
- a bounded `Cell Summary Projection` baseline that future top-level layers may inspect
- the minimum management object family frozen at contract level:
  - `Management Directive`
  - `Delivery Return`
  - `Approval Request`
- explicit ownership and boundary statements for each management object:
  - purpose
  - authority boundary
  - current owning layer
  - what is deferred
- explicit distinction between:
  - product-layer-only structures that may stay in SoloCrew now
  - later `Cognitive_OS` blocker-intake candidates if runtime backing becomes necessary

## Lawful Top-Level Shape

The lawful top-level shape in `v0.4` is intentionally narrow:

- multiple `Cell`s may coexist
- each `Cell` may expose one bounded summary projection
- top-level summary is read-oriented and inspect-oriented
- summary does not silently carry full business-control authority
- summary does not silently become runtime law

This means `v0.4` may freeze the semantics of coexistence and summary before any broad management-plane UX is built.

## Explicit Non-Claims

`v0.4-multi-cell-foundation` explicitly does not claim:

- Secretary interface behavior
- portfolio-level dispatch UX
- growth or BD pack runtime execution
- provider execution
- channel entry or handoff
- broad KPI cockpit behavior
- runtime-complete multi-cell orchestration
- enterprise, team, or multi-tenant behavior

## Product Boundary Rule

SoloCrew may freeze these semantics in product space now because this wave is still about:

- product projection
- product governance
- product management contracts

SoloCrew may not use this wave to invent new mother-runtime law.

If a later multi-cell need requires:

- durable runtime-backed summary generation
- reusable cross-cell orchestration semantics
- shared runtime policy surfaces

that need must be raised later through bounded `Cognitive_OS` blocker intake instead of being silently normalized here.

## Current Classification

| Structure | Current Status In `v0.4` | Later Upstream Trigger |
| --- | --- | --- |
| lawful multi-cell coexistence | product-governance truth in SoloCrew | only if coexistence requires runtime-backed orchestration semantics |
| `Cell Summary Projection` | product projection baseline in SoloCrew | only if summary generation or persistence can no longer stay truthful as downstream product projection |
| `Management Directive` | product management contract in SoloCrew | only if directive state or interpretation must become reusable runtime-backed coordination law |
| `Delivery Return` | product return contract in SoloCrew | only if return semantics must become durable runtime family state rather than bounded product return truth |
| `Approval Request` | product escalation contract in SoloCrew | only if approval coordination must become reusable runtime-backed escalation law |

## Version Semantics Note

In this repo, `v0.4-multi-cell-foundation` is a release-line and governance label.

It is not the same thing as the private workspace version in `package.json`.
So this scope freeze does not claim a new package-distribution version.

## Scope Judgment Rule

If a proposed `v0.4` change mainly serves:

- Secretary UX
- portfolio operating UX
- provider or channel execution
- broad cockpit behavior
- runtime-complete multi-cell orchestration

then it is not a `v0.4` change by default.

If a proposed `v0.4` change makes multi-cell coexistence, summary projection, and management object boundaries more explicit without crossing those later-version lines, it fits this scope.
