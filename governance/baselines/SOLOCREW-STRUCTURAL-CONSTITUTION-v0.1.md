# SOLOCREW-STRUCTURAL-CONSTITUTION-v0.1

## Purpose

This document freezes the post-`v0.1` structural constitution of SoloCrew before any `v0.2` implementation wave begins.

It is a downstream product constitution document.
It does not redefine runtime law in `Cognitive_OS`, and it does not expand protocol law in `MPLP`.

## Product Definition

SoloCrew is a `solo-operator native multi-agent business platform`.

That means:

- the primary user is one operator, not a multi-user enterprise team
- the product is organized around a working business unit, not a generic AI workspace
- the core surface is a persistent crew that helps operate ongoing work
- the product remains a downstream projection over bounded `Cognitive_OS` runtime truth

## Structural Starting Unit

SoloCrew initialization starts from a single `Cell`.

The `Cell` is the smallest valid SoloCrew operating unit.
It is the first structural container that binds together:

- one operator
- one business scope
- one `CEO Orchestrator`
- one native multi-agent crew
- one current objective context
- one execution surface
- one memory and evidence continuity surface

The product must not require portfolio-level organization before the first `Cell` exists.

## Expansion Rule

Portfolio and Secretary expansion are later layers.
They are not day-one initialization requirements.

The structural order is:

1. single operator
2. single `Cell`
3. single `CEO Orchestrator`
4. single native multi-agent crew
5. single current objective and `Cell Console`
6. later management-plane expansion
7. later portfolio and Secretary expansion

This order protects SoloCrew from starting as a fake top-level dashboard with no real operating unit underneath it.

## Downstream Position

SoloCrew remains:

- a downstream product repo
- a projection and app packaging layer
- a product-constitutional surface over bounded runtime truth

SoloCrew does not become:

- a mother-runtime repo
- a protocol repo
- the source of normative runtime object law

The authority order remains:

`MPLP Protocol -> Cognitive_OS -> SoloCrew`

## Structural Principles

### 1. Cell-first before portfolio-first

The first usable SoloCrew version should make one `Cell` real before introducing top-level portfolio management.

### 2. Crew-first before cockpit-first

The product should surface a working crew console before attempting a broad KPI cockpit or enterprise dashboard shape.

### 3. Compile-first before runtime-claim

Structural planning, crew compilation, and management directives may be defined in product space now.
They do not become runtime truth until a bounded runtime blocker is proven and pushed back upward intentionally.

### 4. Constitutional objects before pack breadth

The product must define the minimum constitutional object set for a `Cell` before expanding into optional business packs or metrics packs.

### 5. Honest expansion discipline

If a later SoloCrew structure requires:

- runtime-backed persistence
- runtime-backed orchestration semantics
- shared cross-product protocol promotion

that need must be recorded explicitly instead of being silently invented inside SoloCrew.

## Immediate Structural Non-Claims

This constitution does not claim:

- portfolio management is implemented now
- Secretary is implemented now
- business packs are implemented now
- metrics packs are implemented now
- full multi-cell runtime exists now
- SoloCrew may write structural runtime law into `Cognitive_OS`
