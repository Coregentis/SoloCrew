# SOLOCREW v0.5 Upstream Readiness Check v0.1

## Purpose

This document records what SoloCrew may and may not assume from current
upstream truth before reopening the paused `v0.5` line.

Its job is narrow:

- verify the current `Cognitive_OS` and MPLP state relevant to reopening
- state what SoloCrew may safely consume now
- state what SoloCrew must still not assume
- keep the next wave downstream product semantics only

## Upstream Readiness Basis

Verified current upstream basis:

- `Cognitive_OS` Phase 4 is formally sealed on remote `main`
- bounded MPLP `Confirm` / `Trace` reconstruction/export exists upstream
- unsupported `Context` / `Plan` export is explicitly omitted upstream
- current upstream workflow-truth decision remains `NOT_ADOPTED_UPSTREAM_NOW`
  for SoloCrew packet states
- MPLP current downstream line remains candidate-only and non-promoted

## What SoloCrew May Safely Assume Now

SoloCrew may safely assume:

- the current mother-runtime minimal loop is executable and formally sealed as
  bounded Phase 4 closure
- bounded `Confirm` / `Trace` export posture exists upstream where frozen truth
  authorizes it
- export omission for `Context` / `Plan` is lawful and governed rather than a
  hidden defect
- upstream runtime-private truth remains available only in bounded downstream
  consumption form
- any next SoloCrew wave may remain a product interpretation over those bounded
  upstream surfaces

## What SoloCrew Must Still Not Assume

SoloCrew must still not assume:

- canonical upstream `Context` export exists
- canonical upstream `Plan` export exists
- upstream packet-state workflow truth exists
- upstream has adopted:
  - `staged`
  - `ready_for_cell_review`
  - `returned_for_revision`
  as mother-runtime state-machine law
- MPLP has promoted any of:
  - Cell semantics
  - Portfolio semantics
  - Secretary semantics
  - management-packet semantics
  into protocol law

## Meaning Of Bounded `Confirm` / `Trace` Readiness

The presence of bounded upstream `Confirm` / `Trace` support means SoloCrew may
now lawfully do downstream work such as:

- explanation framing
- rationale visibility
- evidence visibility
- review narration
- bounded portfolio-level summary polish over evidence posture

It does **not** mean SoloCrew may now do:

- approval execution
- rejection execution
- dispatch execution
- workflow-engine behavior
- runtime-authoritative state mutation

## Meaning Of Lawful `Context` / `Plan` Omission

Lawful `Context` / `Plan` omission means:

- SoloCrew must not invent downstream claims that upstream canonical
  `Context` / `Plan` support already exists
- SoloCrew may reference the omission honestly as part of explanatory product
  framing if needed
- any next wave must avoid treating missing canonical `Context` / `Plan` as
  justification for local pseudo-runtime law

## Packet-State Boundary Check

Current upstream still explicitly refuses packet-state adoption.

Therefore the next SoloCrew wave must remain:

- downstream product semantics only
- non-executing
- explanation-first
- posture/rationale/evidence-facing rather than control-facing

## Net Readiness Result

The current upstream state is sufficient for one bounded SoloCrew reopening
wave.

But the next wave must remain:

- downstream
- product-projected
- non-executing
- non-authoritative over runtime or protocol meaning
