# SOLOCREW-UPSTREAM-BLOCKER-LEDGER-v0.1

## Purpose

This ledger records which new SoloCrew structures can remain in product space now, which structures may later require bounded `Cognitive_OS` runtime extensions, and which ideas are only `MPLP` candidate backlog material.

This file does not promote anything upward by itself.

## 1. Can Stay In SoloCrew Product Space Now

These structures may remain in SoloCrew now because they are product-constitutional, projection-owned, or UI/workflow-owned concepts.

| Structure | Why It Can Stay In Product Space Now |
| --- | --- |
| `Cell Charter` | It is a product operating-unit constitution object, not a mother-runtime worker schema. |
| `Delivery Contract` | It defines product delivery expectation, not raw runtime execution law. |
| `Cell Policy Profile` | It defines product operating boundaries, not a generalized upstream runtime policy engine. |
| `CEO Orchestrator Contract` | It is a product contract for how SoloCrew shapes a `Cell`; it does not by itself require a new runtime substrate. |
| `Crew Blueprint` | It is product-owned structural intent for crew shape. |
| `Objective Portfolio` | In current form it is a product planning/ordering object above the active objective surface. |
| `Management Directive` | It is a management-to-cell product object, not a runtime execution envelope. |
| `Cell Summary Card` | It is a management-facing projection object. |
| `Escalation / Approval Request` | It is a product interaction object between management and a `Cell`. |
| `Delivery Return` | It is a product return object, not a runtime event stream. |
| `Business Pack Mount` | As a mount concept it may stay product-side until real runtime mounting needs appear. |
| `Metrics Pack Mount` | As a mount concept it may stay product-side until real runtime/evaluation wiring is needed. |
| `Architecture Planes` | These are product structuring tools, not runtime law. |
| `Version Roadmap` | This is a product governance artifact only. |

## 2. Likely To Require Future Cognitive Runtime Extensions

These items may later need bounded `Cognitive_OS` runtime extensions if SoloCrew wants them to be durable, reusable, and runtime-backed instead of product-only contracts.

They are not blockers for the current docs wave.

| Structure / Need | Why It May Need Runtime Extension Later | Why It Is Not A Blocker Now |
| --- | --- | --- |
| Durable `Cell` constitutional state | If `Cell Charter`, `Delivery Contract`, or `Cell Policy Profile` must become runtime-backed and shared with runtime behavior, they will need lawful persistence/state surfaces. | Current wave only freezes product constitutions. |
| Crew compilation substrate | If the `CEO Orchestrator` must compile into durable runtime worker topology/policy state, SoloCrew will need bounded upstream runtime support. | Current wave only freezes compile contract semantics. |
| Runtime-backed recompile triggers | If recompile conditions must be evaluated automatically inside runtime loops, a reusable runtime trigger surface may be needed. | Current wave only records the conditions conceptually. |
| Durable execution ledger beyond current work-item truth | If SoloCrew later needs a richer ledger than objective/work-item persistence, that goes beyond current stored truth. | Current wave does not require event persistence expansion. |
| Durable memory/evidence anchor beyond bounded preference/memory state | If SoloCrew later needs a stronger evidence substrate, it will likely need upstream support. | Current wave keeps memory/evidence claims narrow and honest. |
| Multi-cell coordination runtime | If multiple `Cell`s share runtime orchestration or resource logic, that likely belongs in bounded upstream runtime extensions. | Single-cell remains the nearer-term mainline. |
| Pack execution mounting | If business packs or metrics packs need runtime mounting and standardized execution hooks, that may require upstream extension. | Current wave only freezes optional mount concepts. |

## 3. MPLP Candidate Backlog Only

These ideas should be treated as candidate backlog material only.
They are not normative protocol content.

| Candidate Idea | Why It Is Only Candidate Material Now |
| --- | --- |
| Generic directive envelope semantics across products | SoloCrew has a product need for `Management Directive`, but there is not yet proof that it should become protocol law. |
| Generic escalation/approval envelope semantics | `Escalation / Approval Request` may later prove reusable, but no promotion case is established here. |
| Generic delivery-return envelope semantics | `Delivery Return` may later become cross-product useful, but it is still product-side now. |
| Generic pack mount descriptor model | `Business Pack Mount` and `Metrics Pack Mount` are still SoloCrew structural concepts, not protocol obligations. |
| Generic compile/recompile vocabulary | `Crew Compiler` concepts may later inform protocol candidate backlog, but they are not protocol law now. |

## 4. Trigger Discipline

SoloCrew should only push something upward when all of the following are true:

1. the product object or behavior can no longer stay truthful as product-only structure
2. a bounded reusable runtime need is clearly exposed
3. the need cannot be solved by lawful downstream projection/app packaging
4. the push target is chosen correctly:
   - `Cognitive_OS` for runtime family needs
   - `MPLP` candidate backlog for possible protocol-level ideas

## 5. Current Judgment

Current judgment for this wave:

- no new blocker requires immediate `Cognitive_OS` modification
- no new idea should be promoted into normative `MPLP` content
- SoloCrew may freeze its structural constitution in product space now and defer upward promotion until later evidence exists
