## 6. V2.3 First Paid Pilot Definition

V2.3 is a bounded design-partner paid pilot planning line for converting the
verified V2.2 local review-only journey into a controlled commercial validation
loop.

V2.3 must remain:

- manual-first
- low-scale
- design-partner-only
- review-only
- non-executing
- non-public-beta
- non-GA
- without automated payment processor unless later authorized
- without provider/channel dispatch
- without marketplace
- without autonomous execution

This line validates whether the V2.2 local journey is useful enough for a
bounded paid pilot. It does not claim broad paid readiness.

## 7. Target Pilot User / Design Partner Profile

Ideal V2.3 pilot user:

- solo founder, protocol builder, AI-product builder, or small technical operator
- has an existing repo or project governance burden
- wants review packet, continuation, and next-action proposal support
- is willing to pay for guided pilot or design-partner access
- accepts manual onboarding and manual payment
- understands the product is not public beta, SaaS GA, or automated execution

The design partner should have a concrete governance/release/project-review
need and enough technical literacy to evaluate local artifacts.

## 8. V2.3 Value Proposition

The pilot user receives:

- a structured project governance packet
- saved workspace continuity
- local review packet export
- founder dashboard continuation
- next-action proposal
- feedback capture path
- possible case-study candidate path

The user pays for guided application of the verified V2.2 local journey to a
real project governance burden. V2.3 validates whether the pain is urgent
enough, whether the generated artifacts are valuable enough, and whether the
manual pilot support burden is sustainable.

V2.2 is sufficient as the technical baseline because it already proves local
workspace continuity, deterministic review packet export, dashboard
continuation, CGOS posture-ref consumption, private-alpha E2E, and negative
boundary gates.

## 15. Pilot Success Criteria

Planning metrics for V2.3:

- 3 to 5 design partners identified for initial pilot consideration
- at least 2 completed project governance packets
- feedback completeness above 80 percent on agreed questions
- at least 1 explicit case-study permission candidate, if appropriate
- time-to-first packet less than 2 business days after manual intake acceptance
- user-reported usefulness at 4 out of 5 or better
- at least 1 repeat usage or continuation signal
- willingness-to-pay or renewal signal recorded for each completed pilot
- support burden tracked per pilot
- boundary incident count of 0 for dispatch, marketplace, autonomy, public beta,
  package publishing, MPLP certification, and MPLP endorsement

These are planning metrics only, not claims that the pilot is ready or sold.
