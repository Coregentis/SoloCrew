# SoloCrew V1.2 RC Known Limitations v0.1

`doc_id: SOLOCREW-V1.2-RC-KNOWN-LIMITATIONS-v0.1`

## A. Purpose

Document current known limitations after V1.2 RC.

## B. Limitations Table

| Limitation | Why it exists | User impact | Future direction |
|---|---|---|---|
| no provider/channel execution | V1.2 RC remains intentionally non-executing | users can review revision candidates but cannot send externally | later product hardening or later product lines only if separately authorized |
| no approve/reject/dispatch/execute | direct control semantics remain out of scope | users see posture and interpretation guards rather than command buttons | later product decisions only if separately authorized |
| no founder queue | queue semantics were intentionally excluded | users should not expect queued founder workflow behavior | possible future product exploration only |
| no live external workflow | V1.2 RC is a bounded internal revision loop | no end-to-end external action flow is available | future product work only if separately authorized |
| no autonomous company operation | the release remains far below autonomy claims | users should not infer self-operating behavior | out of scope for V1.2 RC |
| no protocol certification | the RC line stays below certification language | no certification claim is implied | protocol matters remain upstream-only |
| no Cognitive_OS runtime-private exposure | SoloCrew remains a downstream product consumer | users see projection-safe product fields only | future improvement remains downstream consumption only |
| no MPLP schema/protocol change | this RC does not alter upstream law | users should not expect protocol evolution from this release | future backflow only as evidence or candidate notes |
| no GA/stable release | this is a prerelease line | users should treat V1.2 as bounded RC behavior rather than a stable promise | later release judgment only after separate validation |
| no persistence/queue semantics beyond current bounded projection state | V1.2 RC is centered on review-only packet revision surfaces | no broader durable workflow or queue semantics are introduced | future hardening only if separately authorized |
| evidence gap is not proof/certification | evidence semantics are intentionally bounded below proof claims | users should treat evidence gap copy as explanatory, not conclusive proof | future explanation improvements may clarify this further |
| revision candidate is not approval | revision output remains below decision authority | users should not interpret revised state as approval | future product copy may explain this more clearly |
| return-for-revision is not rejection | return posture remains a bounded revision state | users should not interpret return posture as rejection | future UX explanation only |
| revised packet is not execution | revised packet output remains below execution semantics | users should not interpret packet readiness as execution | future UX explanation only |
